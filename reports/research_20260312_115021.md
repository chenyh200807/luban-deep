# Study Notes: Agentic RAG

## What These Notes Cover
These notes explain the transition from static retrieval to autonomous, tool-using agents that dynamically manage data. You will learn how frameworks like AutoAgent orchestrate specialized sub-agents to solve complex, multi-step queries with higher accuracy than traditional RAG.

## Note 1. Core Mechanics: From Static to Dynamic Retrieval

### What It Means
Agentic RAG replaces simple similarity searches with **Actionable Engines**. Instead of just fetching text, the system uses an autonomous agent to decide which tools to use, how to modify queries, and when the gathered information is sufficient to answer.

> **Definition — Actionable Engine**: An LLM-driven coordinator that interprets natural language tasks and executes them by calling specific tools or sub-agents rather than following a fixed retrieval script.

### How It Works
The agent operates a self-managing file system. For example, if a user asks "How does the new tax law affect EV credits?", the agent doesn't just search once; it follows an iterative loop:

1.  **Query Modification**: Uses `modify_query` to break the prompt into specific sub-questions (e.g., "2024 EV tax credit eligibility").
2.  **Dynamic Storage**: Uses `save_raw_docs_to_vector_db` to convert multi-modal files (.pdf, .docx) into 4096-token chunks for its "memory."
3.  **Verification**: Uses `can_answer` to check if the retrieved data is complete. If not, it loops back to step 1.
4.  **Final Synthesis**: Uses `answer_query` to generate a grounded response.

> 🔑 **Key Takeaway**: Agentic RAG is **iterative**, not linear. It prioritizes data sufficiency over simple keyword matching.

### Common Mistakes
> ⚠️ **Common Mistake**: Treating Agentic RAG as just a "faster" RAG. It is often slower due to multiple LLM calls. The primary pitfall is **infinite loops**, where an agent keeps refining a query without ever settling on an answer due to poorly defined exit conditions.

## Note 2. Frameworks & Multi-Agent Orchestration

### What It Means
Modern agentic systems move away from a single "do-it-all" model toward an **Orchestrator-Worker** model. Frameworks like AutoAgent act as an "Agent Operating System" to coordinate specialized skills.

| Feature | LangChain (Standard Agentic) | AutoAgent (Agent OS) |
| :--- | :--- | :--- |
| **Workflow** | Often requires manual coding/pre-defined paths | Fully automated "on-the-fly" orchestration |
| **Complexity** | Developer-centric | Zero-code, natural language driven |
| **Performance** | High, but limited by static logic | Superior on MultiHop-RAG and GAIA benchmarks |

### How It Works
In a **MultiHop-RAG** scenario (queries requiring multiple data points from different sources), an **Orchestrator** assigns tasks to specialized sub-agents:

*   **Web Agent**: Uses tools like `web_search` and `visit_url` in a secure browser to find real-time news.
*   **Local File Agent**: Converts local spreadsheets or videos into Markdown for long-context analysis.
*   **Coding Agent**: Executes Python in a Docker sandbox for data processing or math.

This division of labor prevents "context pollution"—where irrelevant data from one source confuses the reasoning for another.

### Common Mistakes
> ⚠️ **Common Mistake**: **Over-engineering simple queries**. Using a multi-agent framework for a basic "What is X?" question increases latency and cost without providing a proportional benefit in accuracy.

## Quick Reference

### Key Terms
| Term | Definition |
| :--- | :--- |
| **Actionable Engine** | The LLM-driven core that coordinates tasks and tool usage. |
| **MultiHop-RAG** | Retrieval tasks requiring the synthesis of information from multiple, disconnected sources. |
| **Self-Managing File System** | A system where agents autonomously handle data ingestion, chunking, and vector DB storage. |
| **Orchestrator Agent** | The "brain" that decomposes user requests and delegates them to specialized workers. |

### Top Takeaways
1.  **Iterative Logic**: Agentic RAG uses feedback loops (`can_answer`) to ensure accuracy before responding.
2.  **Specialization**: Sub-agents (Web, Code, File) reduce error rates by focusing on specific domains.
3.  **Dynamic Orchestration**: Systems like AutoAgent outperform static workflows by adapting to the query in real-time.

### Critical Pitfalls
*   **High Latency**: Multiple LLM reasoning steps take significantly more time than a single vector search.
*   **Cost**: Each step in the agentic loop consumes tokens.
*   **Logic Loops**: Agents may get stuck in a "refinement trap" if they cannot find a perfect answer.
