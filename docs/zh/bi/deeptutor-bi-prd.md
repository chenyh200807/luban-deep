# DeepTutor BI PRD

## 1. 文档信息

- 产品名称：DeepTutor BI
- 文档版本：v1
- 更新时间：2026-04-14
- 参考来源：`FastAPI20251222` 项目的 BI 路由、服务和会员运营思路
- 目标：为 DeepTutor 建立一套可服务经营、产品、教学、运营和模型平台的统一 BI 系统

---

## 2. 背景与问题

DeepTutor 当前已经具备丰富的业务能力：

- 多入口：CLI、Web、WebSocket API、Python SDK、小程序、TutorBot
- 多能力：`chat`、`deep_solve`、`deep_question`、`deep_research`、`visualize`、`math_animator`
- 多数据层：会话、消息、回合、知识库、记忆、笔记本、学习画像、会员积分、Langfuse 可观测性

但目前缺少一个统一 BI 层，导致以下问题：

1. 经营层无法快速知道用户增长、留存、活跃、付费和成本结构。
2. 产品层无法判断哪些 capability、tool、知识库和交互路径真正提升了学习效果。
3. 运营层无法快速识别高价值用户、风险用户、续费窗口和学习干预机会。
4. 模型平台无法在一个面板里同时看到质量、成本、时延、模型分布和异常变化。
5. 内容与知识库团队无法衡量哪些知识资产被频繁使用、哪些召回无效、哪些资料值得继续建设。

因此需要建立一套面向 DeepTutor 业务特性的 BI 系统。

---

## 3. 产品目标

### 3.1 核心目标

1. 建立统一经营驾驶舱，让团队随时看到业务健康度。
2. 建立“学习闭环”视角的数据链路，而不只是技术日志。
3. 建立“Agent 闭环”视角的数据链路，衡量 capability、tool、knowledge、TutorBot 的真实效果。
4. 建立会员与用户运营视角的数据链路，支持续费、召回、分层运营。
5. 建立统一指标口径，避免不同团队各算各的。

### 3.2 非目标

1. 当前阶段不追求复杂实时 OLAP 引擎。
2. 当前阶段不追求替代 Langfuse、Prometheus 等专业可观测性产品。
3. 当前阶段不直接做自动化营销系统，只先提供洞察和操作入口。

---

## 4. 目标用户

### 4.1 创始人 / 业务负责人

关心：

- 总活跃、留存、增长、收入、成本、利润空间
- 高价值功能是否被使用
- 业务是否健康，是否有明显异常

### 4.2 产品经理 / 教学产品负责人

关心：

- 学习路径是否顺畅
- 用户会不会从 Chat 升级到 Deep 能力
- 章节掌握度是否提升
- 哪些交互最容易中断、流失、失败

### 4.3 用户运营 / 教学运营

关心：

- 哪些用户有续费机会
- 哪些用户活跃下降
- 哪些用户需要人工干预
- 哪些题单、知识库、TutorBot 对留存有帮助

### 4.4 模型平台 / 工程团队

关心：

- 各模型 token、调用成本、时延、错误率
- 哪些 capability 或 tool 路由成本高
- 哪些知识库召回质量不稳定

---

## 5. 业务北极星与一级指标

## 5.1 北极星指标

建议将 DeepTutor 的北极星指标定义为：

**Weekly Active Learners with Successful Learning Outcome**

即：一周内至少完成一次有效学习闭环的活跃学习者数。

“有效学习闭环”建议定义为满足以下任一条件：

- 完成一次问答并有正向学习事件（如笔记保存、知识点掌握提升、练题完成）
- 完成一次深度能力使用（`deep_solve` / `deep_question` / `deep_research`）且回合成功结束
- 完成一次 TutorBot 驱动的任务闭环

## 5.2 一级指标

1. 活跃：DAU / WAU / MAU / 活跃学习者数
2. 留存：次日、7日、30日留存
3. 学习：学习会话数、有效学习会话率、平均会话深度
4. Agent：能力调用分布、工具成功率、升级路径转化率
5. 质量：成功回合率、错误率、重试率、知识命中率
6. 成本：token、调用次数、成本/回合、成本/有效学习闭环
7. 商业：会员数、续费率、积分消耗、ARPU、LTV 倾向

## 6. 产品范围与信息架构

建议 BI 首页采用“一级模块 + 下钻页”的结构。

### 6.1 模块 A：经营总览

回答的问题：

- 现在整体业务是涨还是跌。
- 活跃、留存、收入、成本、质量是否同时健康。
- 今天/本周有没有需要立即关注的异常。

核心卡片：

- DAU / WAU / MAU
- 活跃学习者数
- 本周有效学习闭环数
- 7 日留存
- 会员收入 / 积分消耗
- 总 token / 总成本 / 成本环比
- 成功回合率 / 异常率

图表：

- 近 30 天活跃趋势
- 活跃与成本双轴趋势
- 能力调用分布
- 入口来源分布（Web / CLI / SDK / WS / MiniProgram / TutorBot）

告警区：

- 成本飙升
- 错误率升高
- 某 capability 成功率下跌
- 某知识库召回命中率显著下降

### 6.2 模块 B：用户增长与留存

回答的问题：

- 用户从哪里来。
- 新用户第一周是否留下。
- 不同入口、不同能力、不同会员层级的留存差异是什么。

核心卡片：

- 新增用户
- 激活用户
- 注册到首个会话转化率
- 首次会话到首次有效学习转化率
- 1/7/30 日留存

图表：

- cohort 留存矩阵
- 渠道新增趋势
- 首周行为漏斗
- 用户分层分布（新手 / 成长期 / 高价值 / 风险）

### 6.3 模块 C：学习行为与转化漏斗

回答的问题：

- 用户从“打开产品”到“形成学习成果”中间卡在哪一步。
- 哪些路径最容易产生成果。
- 哪些用户只是提问，但没有真正学会。

核心漏斗：

1. 打开产品
2. 创建/进入 session
3. 发送首条消息
4. 成功完成首个 turn
5. 使用知识库/工具
6. 进入深度能力
7. 保存到 notebook / 完成练题 / 形成学习结果
8. 次日再次回来

核心卡片：

- 人均会话数
- 人均消息数
- 会话成功率
- 平均会话时长
- 平均消息轮数
- notebook 保存率
- followup 触发率

### 6.4 模块 D：Agent 能力与工具效果

回答的问题：

- 哪些 capability 真有价值。
- tool 是不是帮上忙了，还是只增加成本。
- 用户会不会从浅层对话升级到深层能力。

核心卡片：

- capability 使用分布
- capability 成功率
- capability 单次平均成本
- tool 调用率
- tool 调用成功率
- “chat -> deep_solve” 升级转化率
- “chat -> deep_question” 升级转化率

关键图表：

- capability 漏斗
- tool 成本收益四象限
- tool 使用后 notebook 保存率对比
- tool 使用后 session 深度对比

建议重点分析对象：

- `rag`
- `web_search`
- `reason`
- `brainstorm`
- `paper_search`
- `code_execution`
- `geogebra_analysis`

### 6.5 模块 E：知识库与内容资产

回答的问题：

- 哪些知识库最常用。
- 哪些知识库召回效果最好。
- 哪些知识资产被反复命中。
- 题库、教材、规范、外部资料的效果差异是什么。

核心卡片：

- KB 查询次数
- KB 命中率
- 零结果率
- 被引用文档数
- notebook / deep_question / guide 的知识库带动率

图表：

- 知识库热度排行
- 文档/来源类型占比
- 零命中 query TOP 列表
- 热门知识点与冷门知识点分布

### 6.6 模块 F：成本、质量与可观测性

回答的问题：

- 哪个模型最贵。
- 哪个能力组合的成本收益比最低。
- 哪些错误正在上升。
- 是否存在质量下降但表面活跃正常的问题。

核心卡片：

- 总 token
- 输入/输出 token 结构
- 总成本与单次成本
- 平均响应时延
- 首字节时延
- 回合成功率
- 重试率
- 失败原因分布

图表：

- 模型成本占比
- capability x model 成本矩阵
- 错误趋势图
- success/fail/running 状态趋势图

### 6.7 模块 G：会员、积分与营收

回答的问题：

- 会员规模和结构如何。
- 哪些用户有续费机会。
- 积分是促进学习，还是被动消耗。
- 哪些高价值人群值得更高服务投入。

核心卡片：

- 活跃会员数
- 试用、VIP、SVIP 结构
- 7 天内到期人数
- 自动续费覆盖率
- 积分余额与消耗
- 续费率
- 流失风险人数

关键图表：

- tier 分层趋势
- 到期分布
- 积分收支趋势
- 高风险用户池

### 6.8 模块 H：Learner 360 / TutorBot 360

回答的问题：

- 单个用户目前处于什么状态。
- 最近学了什么、卡在哪、要不要干预。
- 某个 TutorBot 是否真正有产出。

Learner 360 建议包含：

- 基础画像
- 最近会话
- 最近能力使用
- 学习目标与章节掌握度
- notebook 记录
- 积分账本
- 风险标签与推荐动作

TutorBot 360 建议包含：

- bot 活跃度
- 带来的用户数
- 会话成功率
- 学习闭环贡献
- 知识库使用情况
- 成本与 ROI

## 7. 角色权限

### 7.1 管理员

- 查看全局经营、用户、成本、营收、会员、审计
- 导出报表
- 查看用户 360

### 7.2 产品 / 教学负责人

- 查看学习、能力、知识库、质量看板
- 查看脱敏后的用户 360

### 7.3 运营

- 查看增长、会员、风险用户、人群包
- 执行备注、跟进、续费触达

### 7.4 工程 / 模型平台

- 查看成本、时延、质量、模型分布、异常日志

## 8. 关键交互要求

### 8.1 统一筛选维度

所有看板至少支持以下筛选：

- 时间范围
- 入口来源
- capability
- tool
- 知识库
- 用户分层
- 会员层级
- 语言 / 品牌 / 渠道

### 8.2 下钻路径

每个总览指标都应允许下钻，例如：

- DAU -> 用户列表 -> 单用户 360
- capability 成功率 -> 某 capability -> 某模型 -> 失败样本
- KB 零结果率 -> query 列表 -> 对应 session 样本
- 高风险用户 -> 用户 360 -> 最近会话 -> 最近账本

### 8.3 运营动作入口

不是只看不做，BI 页面要能接动作：

- 加运营备注
- 标记高优先级回访
- 导出用户包
- 查看最近 session
- 跳转 notebook / member console / session detail

---

## 9. 成功标准

上线后 4 周内，BI 至少达到以下标准：

1. 团队每周例会统一使用同一套经营总览与学习总览。
2. 所有一级指标具备统一口径，不再出现多个版本。
3. 能在 5 分钟内定位“活跃下跌是增长问题、留存问题、质量问题还是成本问题”。
4. 运营能基于 BI 拉出高风险用户与续费窗口用户。
5. 产品能看出哪些 capability / tool / KB 对学习闭环有提升作用。

---

## 10. 风险与约束

### 10.1 数据分散

当前数据同时存在 SQLite、文件、内存态和外部可观测性中，短期必须接受“先汇总、再统一”的过渡形态。

### 10.2 学习结果定义不统一

DeepTutor 不是单一刷题产品，“什么叫学会了”需要统一定义，否则 BI 很容易只剩点击量。

### 10.3 多入口身份归一

Web、CLI、SDK、MiniProgram、TutorBot 的用户身份和 session 标识需要逐步统一，否则留存和 LTV 会失真。

### 10.4 过度追求复杂

BI 第一阶段应优先回答最核心经营问题，而不是一开始就做完所有明细页。

---

## 11. 建议迭代顺序

### V1

- 经营总览
- 活跃/留存
- capability 分布
- token/成本/错误
- 会员总览

### V2

- 学习漏斗
- Learner 360
- 知识库效果
- 高风险学员人群包

### V3

- TutorBot ROI
- 内容资产 ROI
- 自动化周报 / 告警
- 动作闭环与运营工作台
