# DeepTutor BI 会员后台一体化设计

日期：2026-04-22

## 1. 背景

当前 DeepTutor 已经具备两块真实能力：

1. `/bi`
   已有老板工作台雏形，能聚合经营、质量、会员、TutorBot 等多条 BI 数据线。

2. `/member`
   已有会员控制面雏形，能完成开通、续期、撤销、备注、Heartbeat job 控制、Overlay promotion 等后台动作。

但这两块能力目前仍然存在三个问题：

1. `入口割裂`
   `/bi` 更像展示型 dashboard，`/member` 更像运营工具页。老板看到问题后，不能顺畅落到“该处理哪个会员、该做什么动作”的闭环。

2. `后台力度不够`
   现有会员页虽然能做若干动作，但离“全面会员后台”还有差距，缺少真正的生命周期管理、风险队列、批量操作、学员 360、经营审计和套餐/权益管理视图。

3. `视觉方向不够专业`
   当前 `/bi` 的视觉语言偏重展示感与装饰性，不够接近行业成熟后台那种“信息密度高、层次清楚、动作明确、第一屏就能判断问题”的专业管理界面。

用户目标不是“把 BI 做得更花”，而是让 `/bi` 进化成一个真正可运营、可管理、可留痕、可追责的会员后台总入口，并且整体感受接近旧 FastAPI 项目中更清爽的后台体系。

## 2. 设计结论

本次采用方案：`方案 C：老板工作台版会员后台`

一句话定义：

> 把 `/bi` 升级成统一运营控制面，上半屏服务老板判断，下半屏直接接会员运营和学员 360，把“发现问题 -> 找到会员 -> 执行动作 -> 留痕复盘”收成一条主链路。

核心结论：

- `/bi` 成为会员后台统一入口，默认进入 `老板工作台`
- `/member` 保留兼容路由，但实际与 `/bi` 共享后台能力与核心组件
- 首页第一职责是 `经营判断`
- 首页第二职责是 `运营落地`
- 会员后台不新增第二套学员状态 authority，只聚合现有 `member_console`、learner state、overlay、heartbeat 等现有真相源
- 聊天主链路继续保持单一 `/api/v1/ws`，本次设计不新增专用聊天 transport

## 3. 一等业务事实与单一 authority

### 3.1 一等业务事实

本次设计要维护的唯一业务事实是：

`后台中的会员运营状态，必须能稳定回答“这个会员当前是什么状态、有哪些风险、最近发生了什么、运营可以立刻做什么动作”。`

### 3.2 单一 authority

这条事实的 authority 收口如下：

- `会员运营状态`
  由 `deeptutor/services/member_console/service.py` 唯一负责写入、读取和管理
- `学员学习状态`
  由 learner state 相关服务负责提供
- `Overlay / promotion`
  由 overlay service 提供
- `Heartbeat job`
  由 heartbeat 相关服务提供
- `BI 聚合视图`
  由 `deeptutor/services/bi_service.py` 聚合并展示，不产生第二套会员或学习状态真相

### 3.3 本次明确不做的错误方向

- 不让 `/bi` 成为新的学员状态写入源
- 不在前端维护一套并行会员状态缓存并参与业务决策
- 不新增平行“会员 BI 状态模型”与 `member_console` 争夺 authority
- 不把 TutorBot、教学模式、入口 surface 等概念错误升级为会员后台主身份

## 4. 目标与非目标

### 4.1 目标

- 让老板打开 `/bi` 后 5 秒内看懂经营健康和风险优先级
- 让运营在同一入口内直接完成会员管理动作，而不是在多个页面之间跳转
- 让任意会员都可以下钻到完整 `学员 360`
- 让后台具备批量操作、备注、审计、风险处置和运营动作留痕能力
- 让页面风格更接近成熟 CRM / 会员运营后台，而不是展示型落地页
- 让现有 `/member` 的真实能力被复用并增强，而不是重写一套假后台

### 4.2 非目标

- 不重写整个 `member_console` 存储层
- 不改统一聊天 transport 和 `/api/v1/ws` contract
- 不在本次设计中引入新的支付中台、CRM 中台或复杂工作流引擎
- 不为了“行业顶尖”而叠加无必要的概念、状态机和抽象层
- 不在本次直接承诺多角色复杂权限体系；本次默认管理员后台

## 5. 用户角色与核心任务

### 5.1 老板

关心：

- 今天经营状态是否健康
- 活跃、续费、流失和收入是否异常
- 哪些会员风险最高
- 团队今天最该处理哪几件事

核心任务：

- 识别异常
- 查看趋势
- 打开重点会员
- 分配或触发后续处理

### 5.2 运营 / 班主任

关心：

- 哪些会员需要续费、召回、备注或干预
- 某个会员当前的生命周期、学习状态、最近记录和可执行动作

核心任务：

- 搜索和筛选会员
- 开通 / 续期 / 撤销会员
- 管理备注与运营记录
- 处理 heartbeat / overlay 相关问题
- 批量执行运营动作

### 5.3 教研 / 管理者

关心：

- 学员学习风险和教学效果摘要
- TutorBot 运行状态是否影响付费用户体验

核心任务：

- 通过会员 360 看到学习面风险
- 通过 BI 异常摘要定位系统层面问题

## 6. 信息架构

本次采用统一入口结构：

- `老板工作台`
- `会员运营`
- `学员 360`
- `经营审计`

页面结构约束：

- `/bi` 默认进入 `老板工作台`
- `会员运营` 是完整表格工作区，不再只是摘要卡片
- `学员 360` 以右侧抽屉或详情面板方式下钻，不强制离开当前工作流
- `经营审计` 承接操作日志、关键事件和导出
- `/member` 路由继续保留，但跳转或复用到同一后台组件树

## 7. 页面结构设计

## 7.1 顶部工作台头部

顶部采用轻量后台头部，而不是大 Hero。

包含：

- 标题：`DeepTutor 会员后台`
- 当前区域名：`老板工作台 / 会员运营 / 学员 360 / 经营审计`
- 时间窗口切换：`7 天 / 30 天 / 90 天`
- 主动作：`刷新`
- 辅动作：`导出`
- 全局筛选：`会员等级 / 风险等级 / 来源 / 是否即将到期`

要求：

- 高度克制
- 第一屏优先留给数据与行动，不给装饰文案
- 所有全局筛选都可复用于老板工作台和会员列表

## 7.2 老板工作台

目标：先判断业务，再进入动作。

第一屏结构固定为：

1. `经营 KPI 条`
2. `风险与待处理队列`
3. `经营主趋势`
4. `重点会员与即将到期名单`

### 经营 KPI 条

固定展示：

- 总会员数
- 付费会员数
- 即将到期人数
- 7/30 天活跃会员数
- 续费率
- 点数收入或权益包销售额

要求：

- 每个指标要有环比或趋势提示
- 风险型指标要能直接跳入队列或筛选后的会员列表

### 风险与待处理队列

显示：

- 今天到期会员
- 3 天内到期会员
- 高风险沉默会员
- Heartbeat 异常会员
- Overlay promotion 待确认会员
- 高投诉或备注预警会员

每条都必须可点击，并把过滤条件直接带入会员运营区。

### 经营主趋势

优先展示：

- 新增会员趋势
- 续费趋势
- 活跃趋势
- 流失趋势

不展示复杂的工程内部图层混合图。

### 重点会员名单

包括：

- 高价值会员
- 即将流失会员
- 最近升级会员
- 最近有连续异常行为的会员

每个条目都能直接打开 `学员 360`。

## 7.3 会员运营

目标：成为真正可操作的会员管理工作区。

主布局采用：

- 左侧筛选与保存视图
- 中间会员表格
- 右侧详情抽屉或侧栏

### 会员表格字段

必须包含：

- 会员姓名 / user_id
- 手机号
- 会员等级
- 状态
- 来源 / 分群
- 最近活跃时间
- 到期时间
- 风险等级
- 点数余额
- 当前焦点

### 筛选维度

必须支持：

- 搜索：姓名 / 手机号 / user_id
- 会员状态：试用 / 正常 / 即将到期 / 已过期 / 已撤销
- tier：trial / vip / svip 等
- 风险等级：low / medium / high
- 来源 / 分群
- 最近活跃区间
- 到期区间
- auto renew
- 是否存在 heartbeat job
- 是否存在 overlay promotion 待处理项

### 批量操作

必须支持：

- 批量开通
- 批量续期
- 批量撤销
- 批量打标签或备注
- 批量导出当前筛选结果

约束：

- 批量操作必须写入审计日志
- 批量操作结果必须回显成功 / 失败明细

## 7.4 学员 360

目标：打开一个会员后，后台能直接回答“这个人怎么了、该怎么运营”。

结构分为 6 个区块：

1. `基础信息`
2. `会员权益`
3. `学习画像`
4. `TutorBot / Overlay / Heartbeat`
5. `运营记录`
6. `建议动作`

### 基础信息

展示：

- user_id
- display_name
- 手机号
- 注册时间
- 来源 / 分群
- 当前会员状态

### 会员权益

展示：

- tier
- expire_at
- auto_renew
- points_balance
- 最近充值或续费摘要

### 学习画像

展示：

- 最近活跃时间
- 学习天数
- focus_topic / focus_query
- 章节掌握度摘要
- review_due
- difficulty_preference
- explanation_style

### TutorBot / Overlay / Heartbeat

展示：

- 当前 bot overlay 摘要
- promotion 待处理项
- heartbeat job 状态与节奏
- 最近对话或最近活跃摘要

### 运营记录

展示：

- notes
- 审计摘要
- 最近运营动作

### 建议动作

根据现有数据直接给出后台动作入口：

- 开通 / 续期 / 撤销
- 添加备注
- 暂停 / 恢复 heartbeat
- 执行 overlay promotion
- 调整学习目标与偏好

## 7.5 经营审计

目标：把后台变成可追责的管理系统，而不是黑箱。

展示：

- 操作日志列表
- 操作人
- 操作对象
- 操作前后摘要
- 时间
- 原因
- 支持筛选和导出

要求：

- 所有会员关键动作都必须进入审计
- 审计区不承载编辑，只承载查询、筛选和导出

## 8. 后端设计

## 8.1 总体原则

优先扩展现有：

- `deeptutor/api/routers/member.py`
- `deeptutor/services/member_console/service.py`
- `deeptutor/api/routers/bi.py`
- `deeptutor/services/bi_service.py`

不新造第二套餐层。

## 8.2 member API 增强

需要补齐或增强以下能力：

- `member list`
  增加来源、活跃区间、到期区间、heartbeat / overlay 风险等筛选
- `member dashboard`
  增加经营级摘要，供老板工作台复用
- `member 360`
  增加最近充值摘要、学习风险摘要、推荐动作摘要
- `batch actions`
  提供批量开通、续期、撤销、备注、导出
- `audit log`
  提供可前端消费的分页、筛选、导出能力

## 8.3 BI API 增强

BI 侧只做聚合增强：

- 输出老板工作台需要的风险队列
- 输出重点会员名单
- 输出统一经营 KPI 与趋势摘要

约束：

- 不在 BI API 中写会员状态
- 不复制 member_console 的写接口

## 9. 前端设计

## 9.1 风格方向

采用“清爽高密后台”风格：

- 米白 / 石墨灰为基础
- 深棕或暖灰作为品牌强调色
- 收益、风险、告警使用功能色
- 以表格、分栏、状态标签、细边界为主
- 减少大卡片、大块渐变和展示型文案

## 9.2 布局原则

- 第一屏只放判断和动作，不放长文案
- KPI 用条状或紧凑卡，不做宣传式大面板
- 会员列表是后台主工作区，必须信息密度足够
- 学员 360 采用右侧抽屉或固定详情栏，保证“边看边管”

## 9.3 交互原则

- 从老板工作台点风险项，直接落到筛选后的会员运营区
- 从会员表格点人，直接打开学员 360
- 从学员 360 执行动作后，局部刷新，不整页抖动
- 失败提示局部承载，不使用整页大告警横幅覆盖主界面

## 10. 实施阶段

### Phase 1：统一工作台骨架

- 重构 `/bi` 为统一后台结构
- 接入老板工作台、会员运营、学员 360、经营审计四大区
- 保留 `/member` 路由兼容

### Phase 2：会员运营能力补齐

- 扩展筛选维度
- 补批量操作
- 补风险队列落地跳转
- 补更完整的学员 360

### Phase 3：经营与审计收口

- 审计页面前端化
- 导出能力补齐
- 老板工作台的趋势、重点会员和经营摘要收敛

### Phase 4：高级运营能力

- 套餐 / 点数包管理入口
- 保存视图
- 更丰富的风险动作模板

## 11. 验收标准

- 老板打开 `/bi` 后，5 秒内能看到经营健康、风险队列和重点会员
- 运营在同一入口内可完成开通、续期、撤销、备注、heartbeat、overlay 处理
- 任一会员都能打开学员 360，并看到学习画像与运营动作
- 所有关键动作都有审计记录
- 页面风格明显比当前版本更清爽、更像专业后台，而不是展示页
- 本次改动不新增新的聊天 transport，不破坏 `/api/v1/ws` 单一入口

## 12. 风险与约束

### 12.1 数据完整性风险

部分老板工作台指标可能依赖现有 `member_console` 与 `bi_service` 的字段完备度。若部分统计尚未稳定，需要先通过最小聚合满足第一版，而不是引入新状态源。

### 12.2 页面复杂度风险

如果把老板视图、运营视图和学员 360 全塞进同一层，容易再次变成大杂烩。实现时必须坚持“老板工作台负责判断，会员运营负责执行，学员 360 负责下钻”三层边界。

### 12.3 authority 漂移风险

如果为了前端方便把更多状态拼成前端本地真相，会重新制造 competing authorities。实现时必须坚持“读现有真相源、聚合展示、局部写回 member_console”的约束。

## 13. 设计总结

这次不是把 BI 页面继续做厚，而是把 DeepTutor 的会员后台真正立起来。

最终形态不是“一个更漂亮的看板”，而是：

- 上层能让老板判断经营
- 中层能让运营管理会员
- 下层能让管理者下钻到学员 360 并立即执行动作

并且整个系统仍然遵守现有 DeepTutor 的 contract 边界，不额外长出第二套会员或聊天 control plane。
