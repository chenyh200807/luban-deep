# DeepTutor BI 老板工作台版设计

日期：2026-04-21

## 1. 背景

当前 DeepTutor 的 `/bi` 页面已经具备基础数据能力，但存在两个核心问题：

1. `界面方向偏了`
   当前页面更像一个偏“Command Deck 展示页”的综合工作台，视觉过重，Hero、渐变、深色块和装饰性留白占用了第一屏预算，不够像一个老板日常查看的经营 BI。

2. `功能内容偏了`
   首页把 TutorBot、能力、工具等系统内部视角抬得过高，老板真正关心的经营健康、会员风险、收入趋势、留存和待处理事项没有成为绝对主线。

用户明确要求新版本在整体感受上更接近 `/Users/yehongchen/Documents/CYH_2/Markzuo/FastAPI20251222/static/bi/` 的清爽 dashboard，而不是继续沿当前“厚重指挥舱”方向堆内容。

与此同时，当前 BI 还暴露出一个交互层问题：页面在部分接口或装配失败时，会直接出现“部分 BI 接口未完全加载 / Load failed”的显性告警，造成“功能不可用”的感知。即使后端大部分接口已可返回数据，前端也没有把失败限制在局部模块，而是把异常扩大成首页一级噪声。

本设计的目标，是把 `/bi` 重新定义为一个真正服务老板/运营视角的“老板工作台”，同时收口首页的数据装配、异常提示和降级行为。

## 2. 设计结论

本次采用方案：`老板工作台版`

一句话定义：

> 用清爽、克制、经营优先的 BI 首页，替代当前以 TutorBot 和展示性视觉为中心的页面结构。

核心结论如下：

- `/bi` 首页的唯一主角改为 `经营健康`
- 首页第一屏固定采用 `经营 KPI + 经营主图 + 混合待处理区`
- `会员运营 / 经营健康 / 教学质量 / TutorBot 运行态` 作为二级内容结构保留
- TutorBot、能力、工具、知识库等工程/产品内部视角一律下沉，不再抢占首页主叙事
- 首页异常提示收敛为 `局部降级 + 轻提示`，不再用大面积告警横幅破坏整体体验

## 3. 目标与非目标

### 3.1 目标

- 让老板打开 `/bi` 后 3 秒内能判断业务健康
- 让第一页先回答：
  - 收入怎么样
  - 活跃怎么样
  - 付费和续费怎么样
  - 会员风险在哪里
  - 当前最需要处理什么
- 让页面样式明显更接近旧 FastAPI BI 的干净 dashboard，而不是继续保留重 Hero 和展示型质感
- 让首页内容更像 `经营工作台`，不是 `系统内部控制台`
- 让单个接口失败只影响对应模块，不把首页整体打成“不可用”

### 3.2 非目标

- 不把首页做成完整 CRM / 会员运营后台
- 不在首页直接放复杂的运营编辑表单、审计大表和后台管理控件
- 不新增新的 BI API 合同，不重新设计 `/api/v1/bi/*` 后端体系
- 不为了贴近旧项目而机械复刻其所有页面和历史结构
- 不在本次设计中扩展新的 WebSocket、专用 BI transport 或二套数据 authority

## 4. 设计原则

### 4.1 老板视角优先

首页只展示老板需要用于判断和决策的信息：经营结果、趋势、风险、待处理事项。凡是更偏工程、教研或内部诊断的信息，都应下沉。

### 4.2 清爽优先于戏剧化

视觉上减少大面积深色块、重渐变、夸张 Hero、厚重阴影和装饰性提示，恢复 dashboard 的稳定感和连续阅读体验。

### 4.3 首页先判断，再下钻

首页不是终点，而是老板判断“现在是否健康、问题在哪、下一步点哪”的第一屏。所以首页要把结论和入口放在一起，而不是先铺大量散乱模块。

### 4.4 局部失败不放大

前端必须把 API 失败限制在对应卡片或对应区块。首页不应因为一个子请求失败就呈现“整页不可用”的感知。

### 4.5 复用现有 authority

本次只重做页面结构、前端装配和异常承载方式，不新增后端平行 contract。BI 真相仍来自当前 `/api/v1/bi/*`。

## 5. 页面结构

## 5.1 顶部控制条

首页顶部不再使用大 Hero，而改为轻量工作台头部。

包含：

- 标识：`Luban BI / 老板工作台`
- 全局时间窗口：`7 天 / 30 天 / 90 天`
- 动作：`刷新`
- 轻量筛选入口：`筛选`

要求：

- 控制条高度明显低于当前版本 Hero
- 不再使用大副标题、厚重材质和强展示型视觉
- 筛选器默认折叠，不占用第一屏核心区域

## 5.2 第一屏信息层级

首页第一屏固定为以下顺序：

1. `5 张经营 KPI 卡`
2. `1 张经营主趋势图`
3. `右侧混合待处理区`
4. `会员分层 / 留存 / 渠道` 三块结构信息
5. `重点会员名单 / 告警与备注`

这条顺序是硬约束，不允许被 TutorBot、工具效率、模型成本等内容挤占。

## 5.3 二级内容结构

首页下方或二级 tab 保留以下 4 条内容线：

1. `会员运营`
2. `经营健康`
3. `教学质量`
4. `TutorBot 运行态`

这些内容是下钻线，不是首页第一视觉中心。

## 6. 功能内容定义

## 6.1 经营 KPI 区

### 展示内容

- 总营收 / 当期营收
- 日活 / 周活
- 付费会员数
- 续费率
- 高风险会员数

### 不展示内容

- 模型名
- TutorBot 运行细节
- 工具调用细目
- 工程内部拆账项

### 实现说明

优先从现有：

- `overview`
- `members`
- `cost`

中组装出经营卡片。若后端字段不完全对应，允许前端在现有合同之上做最小映射，但不新增平行接口。

## 6.2 经营主图

### 展示内容

- 营收趋势
- 活跃趋势
- 付费趋势

### 不展示内容

- 多条工程指标混合叠加
- 高复杂度双轴图
- 与老板无关的系统内部折线

### 实现说明

主图必须服务“经营变化判断”，不是展示图表能力。若当前数据源不足以同时稳定提供三条曲线，优先保证营收和活跃，再补付费。

## 6.3 混合待处理区

### 展示内容

- 收入异常
- 活跃下滑
- 高风险会员
- 即将到期会员

### 形式

- 固定 4 到 6 条
- 每条都是一句明确可执行的事项
- 每条都能跳转到对应下钻区块

### 不展示内容

- 抽象口号式洞察
- 需要二次解释才能理解的工程告警

## 6.4 会员运营区

### 展示内容

- 会员分层
- 流失风险分布
- 重点会员名单
- 快到期名单

### 不展示内容

- 直接把运营编辑器放在首页
- 大量管理表单、批量操作和审计面板

### 实现说明

首页只承载“看谁值得关注”，详细操作下沉到二级区块或抽屉。

## 6.5 经营健康区

### 展示内容

- 留存漏斗
- 渠道 / 入口来源
- 新老会员结构

### 不展示内容

- 复杂 cohort 大矩阵直接砸在首页主区域

### 实现说明

留存和渠道需要以老板能看懂的摘要方式出现，而不是研究用原始细表。

## 6.6 教学质量区

### 展示内容

- 学习效果摘要
- 知识库命中概况
- 异常波动

### 约束

该区的表述必须是老板能理解的业务语言，不应充满工具、模型和系统内部术语。

## 6.7 TutorBot 运行态区

### 展示内容

- 状态摘要
- 异常摘要
- 必要时的告警入口

### 下沉内容

- TutorBot 排行
- 最近消息
- 工具效率
- 能力排行
- 知识库副面板

### 原因

这些内容对于产品/工程有价值，但不应该再占首页中心视觉预算。

## 7. 视觉设计

### 7.1 风格方向

目标视觉应接近旧 FastAPI BI 的清爽 dashboard：

- 白底或浅底为主
- 轻阴影
- 轻边框
- 中性留白
- 稳定信息密度

### 7.2 本次明确删除

- 大 Hero
- 大面积铜棕深色块
- Command Deck 戏剧化标题
- 首页默认全展开筛选器
- 大横幅式警告区

### 7.3 本次保留

- DeepTutor / Luban 的品牌命名
- 适度的暖色品牌点缀
- 轻量卡片、状态色、趋势提示

## 8. 交互设计

## 8.1 时间窗口

- 仍保留 `7 / 30 / 90 天`
- 作为全局时间窗口，统一作用于老板首页各模块
- 切换时刷新首页主数据，不重置当前页面结构

## 8.2 筛选器

- 首页默认折叠
- 只有需要下钻时才展开
- 首页第一屏不能被筛选器占据主空间

## 8.3 告警与失败提示

### 当前问题

当前页面出现“部分 BI 接口未完全加载 / Load failed”时，会直接把失败升级成首页一级噪声。

### 新设计

- 若单个模块失败，只在该模块内显示轻量空态或局部错误文案
- 首页顶部仅在“核心经营模块不可用”时显示轻提示
- 轻提示文案要具体，例如：
  - `经营趋势暂未返回，已展示其余经营数据`
  - `会员样本加载失败，可稍后重试`

### 明确禁止

- 用通用大黄条覆盖式提示破坏第一屏主叙事
- 因单接口失败把整页打成“降级展示”

## 8.4 下钻路径

每个首页区块都需要明确下钻去向：

- KPI / 主图 -> 经营健康
- 风险会员 / 即将到期 -> 会员运营
- 学习异常 / 命中异常 -> 教学质量
- 运行状态 / 告警入口 -> TutorBot 运行态

## 9. 数据装配与错误处理

## 9.1 前端装配

前端需要从现有 `/api/v1/bi/*` 中重新映射老板首页所需内容。优先使用：

- `overview`
- `active-trend`
- `retention`
- `members`
- `cost`
- `anomalies`

其余接口按二级区块使用。

## 9.2 错误处理要求

- `loadBiWorkbench()` 不得再把任一子接口失败直接提升成首页统一灾难提示
- 首页模块必须支持：
  - `正常态`
  - `无数据空态`
  - `局部失败态`
- 前端必须区分：
  - `接口失败`
  - `接口成功但数据为空`

空数组和 0 值不能被误判为失败。

## 9.3 部署验证要求

由于用户当前反馈“用不了”，实现后必须做真实验证，而不只是本地静态审阅：

- 本地运行 `/bi`
- 逐项验证首页主模块加载
- 阿里云重新部署后验证公网 `/bi`
- 验证关键接口实际可达：
  - `overview`
  - `active-trend`
  - `members`
  - `retention`
  - `anomalies`

## 10. 测试与验收

## 10.1 设计验收

页面通过以下标准才算达成目标：

- 第一眼更像经营 dashboard，而不是 Command Deck 展示页
- TutorBot 不再是首页主角
- 第一屏能自然回答经营是否健康
- 页面明显比当前版本更清爽、更稳、更接近 FastAPI 旧版

## 10.2 功能验收

- 首页 5 张经营 KPI 正常可见
- 经营主图正常渲染
- 混合待处理区可见且每项可下钻
- 会员分层 / 留存 / 渠道区块正常显示
- 单个子接口失败时，首页仍能继续使用

## 10.3 自动化与回归

实现阶段至少补以下验证：

- `/bi` 首页主结构存在性测试
- 经营 KPI 与主图渲染测试
- 局部失败降级测试
- TutorBot 细项不再抢首页主区测试
- 公网部署后的真实访问验证

## 11. 实现边界

本次实现只围绕以下范围展开：

- `/bi` 首页结构重做
- 首页内容重组
- 轻量视觉系统替换
- 局部错误承载方式重做
- 二级区块优先级调整

以下内容不在本次范围：

- 新 BI 后端 contract
- 完整会员运营后台
- TutorBot 深度运维页面
- 新 CRM / 审计系统

## 12. 设计后的实现顺序

建议实现顺序如下：

1. 先收首页信息架构和视觉骨架
2. 再重组经营 KPI、主图、混合待处理区
3. 再下沉 TutorBot 与工程视角内容
4. 再收口局部失败与空态逻辑
5. 最后补本地和公网验证
