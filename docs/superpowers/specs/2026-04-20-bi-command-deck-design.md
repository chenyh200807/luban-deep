# DeepTutor BI Command Deck Design

日期：2026-04-20

## 1. 背景

当前 DeepTutor 的 BI 页面已经具备基础数据能力，能够展示总览、趋势、留存、TutorBot、能力、知识库、会员、成本以及 Learner 360 等数据块，但整体呈现仍然更接近“单页工作台”。

用户希望新版本 BI 在设计样式和功能组织上接近 `/Users/yehongchen/Documents/CYH_2/Markzuo/FastAPI20251222/` 中的旧 BI 页面，尤其是其“指挥舱 / Command Deck”式的信息架构、模块分区和视觉层级；同时保留 DeepTutor 当前系统的品牌气质与现有后端合同，不做一次性重建旧系统后台。

本设计的目标是把当前 DeepTutor BI 从“数据工作台”收敛为“指挥舱”，优先完成首页重组、主叙事收口、交互稳定化和降级策略明确化。

## 2. 设计结论

本次采用以下组合方案：

- 信息架构：对齐旧仓 BI 的指挥舱结构，而不是继续维持当前单页长卷的弱分区结构。
- 视觉气质：采用 DeepTutor 自身的铜棕暖色品牌方向，不直接复刻旧仓蓝白工业风。
- 实现策略：最大化复用当前 `/api/v1/bi/*` 数据合同，不新增一套新的 BI API 体系。

一句话定义：

> 用旧仓的指挥舱骨架，承载 DeepTutor 自己的品牌气质和当前 BI 数据能力。

## 3. 目标与非目标

### 3.1 目标

- 让 BI 首页第一眼明显更接近旧仓 `/bi` 的“指挥舱”观感。
- 让页面主叙事聚焦为四条主线：
  - 总览
  - 质量/稳定性
  - 会员经营
  - TutorBot/能力
- 让现有 DeepTutor BI 数据块有清晰归属，不再并列散放。
- 保证接口部分失败时，页面仍能稳定展示，不出现整屏空白或主分区消失。
- 保留当前 Learner 360 明细能力，但下沉到 drill-down 抽屉，不抢首页主叙事。

### 3.2 非目标

- 不在第一版中补齐旧仓全部会员管理、审计、运营编辑、评测和 observability 深功能。
- 不新增新的 BI WebSocket、专用聊天入口或新的控制面 contract。
- 不为了贴近旧仓而重构后端 BI 服务的数据模型。
- 不把首页升级成一个完整 CRM/运营后台。

## 4. 设计原则

### 4.1 指挥舱优先，不是信息堆砌

首页要优先回答四个问题：

- 系统整体怎么样？
- 当前稳不稳？
- 会员与经营面怎么样？
- 主业务 AI 能力跑得怎么样？

任何无法直接服务这四个问题的内容，都不应占据首页一级主区。

### 4.2 复用现有 authority

本次设计不改变当前 BI 的后端 authority。前端只重组页面，不另起一套与现有 `/api/v1/bi/*` 并行的新数据真相。

### 4.3 首页稳定优先于局部完整

主分区必须稳定存在。某个子模块无数据时，展示降级态，而不是隐藏整个分区或让页面断裂。

### 4.4 DeepTutor 品牌保留

整体布局、控制条、Tab 层级、卡片权重对齐旧仓，但配色、材质、品牌语气保持 DeepTutor 当前暖色铜棕方向。

## 5. 页面结构

## 5.1 顶部 Hero

首页第一屏采用指挥舱 Hero 结构，包含：

- 标题区：`DeepTutor BI Deck`
- 副标题：经营、质量、会员、TutorBot 四条主线的一体化指挥舱
- 全局时间窗口：`7 天 / 30 天 / 90 天`
- 全局动作：`刷新`、`导出`
- 当前状态摘要：最近同步时间、当前窗口、当前筛选
- 一级 Tab：`Overview / Quality / Member Ops / TutorBot`

设计要求：

- Hero 在视觉上承担页面控制台入口的角色。
- Hero 与一级 Tab 要有明确层级，不退化成普通表头加几个按钮。
- 保持 DeepTutor 铜棕渐变和暖色背景，不切换为旧仓冷色工业风。

## 5.2 一级主分区

首页保留且仅保留以下四个一级主分区：

1. `总览`
2. `质量/稳定性`
3. `会员经营`
4. `TutorBot/能力`

这些分区通过一级 Tab 切换主内容区展示。页面保持单页面受控主视图，不引入多层嵌套路由。

## 6. 各主分区设计

## 6.1 总览

### 作用

回答“系统整体现在怎么样”。

### 模块组成

- 6 张核心 KPI 卡
- 总趋势图
- 异常摘要
- 经营快照副面板
- 会员风险摘要副面板
- TutorBot 状态摘要副面板

### 数据映射

优先使用：

- `overview`
- `active-trend`
- `anomalies`

必要时从以下数据中抽取少量摘要卡：

- `cost`
- `members`
- `tutorbots`

### 约束

- 不在该区铺开完整知识库、Learner 360 或大表格。
- 该区必须承担第一屏“抬头”能力，而不是继续当作普通数据列表容器。

## 6.2 质量/稳定性

### 作用

回答“系统稳不稳，哪里在坏”。

### 模块组成

- 成功率/健康摘要卡
- 趋势波动面板
- 异常列表
- 质量摘要
- 数据完整性提示

### 数据映射

优先使用：

- `active-trend`
- `anomalies`
- `feedback`
- `overview.alerts`（若存在）

### 约束

- 不伪造旧仓才有、而当前 DeepTutor 后端并未稳定提供的 observability stack 深面板。
- 若暂无完整 observability 明细，也必须保留该主分区，并用现有趋势和异常数据承接主叙事。

## 6.3 会员经营

### 作用

回答“会员质量、留存和经营风险在哪里”。

### 模块组成

- 会员核心卡
- tier 分布
- risk 分布
- 留存热力图
- 样本用户列表
- 点击样本后的 Learner 360 抽屉

### 数据映射

直接复用：

- `members.cards`
- `members.tiers`
- `members.risks`
- `members.samples`
- `retention`
- `learner/{user_id}`

### 约束

- 首页只展示“谁值得关注”和“总体经营状态”。
- 详细用户视图走抽屉，不跳新页面，不把首页做成 CRM 大后台。

## 6.4 TutorBot/能力

### 作用

回答“主业务 AI 能力跑得怎么样”。

### 模块组成

- TutorBot 核心指标
- TutorBot 排行
- 状态分布
- 最近活跃
- 最近消息摘要
- 能力表现排行
- 工具效率

### 数据映射

直接复用：

- `tutorbots`
- `capabilities`
- `tools`

### 约束

- 这里要形成一条完整业务线，不再把 TutorBot、capability、tools 作为多个松散区块平铺。

## 7. 下沉内容策略

以下内容保留，但不做首页一级主分区：

### 7.1 知识库

使用 `knowledge` 数据，放入 `TutorBot/能力` 分区的副面板或折叠区。

### 7.2 成本

使用 `cost` 数据，放入 `总览` 的经营快照或 `会员经营` 的副卡，不单独占一个一级主区。

### 7.3 Learner 360

不在首页直接展开；点击会员样本后，以右侧抽屉方式展示。

## 8. 交互设计

## 8.1 一级 Tab

- 一级 Tab 控制主内容区视图。
- 切换 Tab 不跳新页面。
- 切换 Tab 不重置全局时间窗口。
- Tab 的职责是切换主叙事，不是做额外筛选器。

## 8.2 全局时间窗口

- `7 天 / 30 天 / 90 天` 为全局窗口。
- 窗口切换统一作用于首页主数据。
- 窗口状态在 Hero 区显式展示。

## 8.3 刷新与导出

- `刷新` 保留在 Hero 右上角，作为控制台主动作。
- `导出` 继续使用当前 JSON 导出能力。
- 两者视觉权重要明显高于普通筛选按钮。

## 8.4 Learner 360 抽屉

- 点击会员样本打开右侧抽屉。
- 抽屉展示 Learner 360 明细。
- 抽屉不改变当前 Tab，不触发页面跳转。

## 9. 降级与错误处理

## 9.1 主分区稳定存在

若某个接口失败：

- 不隐藏整个主分区
- 不让首页整屏报错
- 不让一级 Tab 失效

而是：

- 保留主分区骨架
- 在缺失模块位置显示明确降级提示
- 继续渲染其余已加载模块

## 9.2 具体降级要求

- `总览`：单个接口失败时，其他 KPI 和趋势继续显示。
- `质量/稳定性`：即使没有完整 observability 细项，也必须展示趋势和异常摘要。
- `会员经营`：没有样本用户时，仍显示 tier、risk、retention。
- `TutorBot/能力`：没有 recent messages 时，仍显示 cards、ranking、status。

## 10. 实现边界

第一版允许的改动：

- 前端信息架构重组
- 视觉风格切换到本设计的 B 方案
- 现有 BI 数据块重新映射与重组
- 空态、降级态、抽屉交互收口

第一版不做：

- 新增旧仓式会员运营编辑器
- 新增新的 BI API 路由族
- 新增专用聊天或运营 WebSocket
- 强行补齐旧仓全部 ARR、AAE、审计、评测、observability 深功能

## 11. 验收标准

以下条件同时满足时，视为设计达标：

- 页面第一眼明显更接近旧仓 BI 指挥舱，而不是当前单页工作台。
- 首页主叙事明确收敛为：
  - 总览
  - 质量/稳定性
  - 会员经营
  - TutorBot/能力
- 时间窗口和一级 Tab 切换时，页面结构稳定，无大面积抖动或空白。
- 接口部分失败时，页面仍能以降级态可用。
- 点击会员样本能稳定打开 Learner 360 抽屉。
- 不新增新的后端 contract，只基于现有 `/api/v1/bi/*` 重组。

## 12. 实施建议

推荐的实现顺序如下：

1. 先重组首页 Hero 和一级 Tab 骨架。
2. 再把当前页面的数据块映射到四个主分区。
3. 补齐降级态和空态。
4. 最后收口视觉层级、铜棕品牌样式和细节动效。

该顺序的原因是：先稳定结构，再补视觉；先稳定叙事，再补细节。
